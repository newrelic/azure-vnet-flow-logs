E2E Test Plan: Azure VNet Flow Logs Forwarder

Objective

Validate the complete pipeline from Azure VNet flow log generation through to New Relic ingestion, ensuring logs are correctly captured, processed, and delivered with no data loss.

Cost & Safety Guardrails

  • Use smallest VM size (Standard_B1s)
  • Auto-delete resource group after max test duration (30 min hard limit)
  • Storage lifecycle policy: delete blobs after 1 day
  • Tag all resources with purpose=e2e-test and created-by=ci
  • Never hardcode secrets in test scripts — use Key Vault or CI secret store

Test Script Structure

Modular shell scripts (inspired by aws-unified-lambda e2e pattern):

FilePurpose
e2e-tests/common-scripts/test-configs.cfgAll test parameters, resource names, NR query constants, retry settings
e2e-tests/common-scripts/resource-scripts.shAzure resource creation/validation helpers (VNet, VM, storage, Event Hub)
e2e-tests/common-scripts/stack-scripts.shARM template deploy, validate deployment status, capture failure reasons, delete stack with exponential backoff
e2e-tests/common-scripts/logs-scripts.shNerdGraph API log query, exponential backoff retry, attribute validation, entity synthesis checks
e2e-tests/vnet-flow-log-test.shMain test orchestrator — calls infrastructure setup, traffic gen, deployment, NR validation, teardown

Key Patterns from aws-unified-lambda

  • Exponential backoff polling: Initial sleep 30s, double each retry, cap at 180s, max 5 retries when waiting for NR logs
  • NerdGraph API for log verification: Use curl to https://api.newrelic.com/graphql with NRQL queries
  • Deployment failure capture: On stack deploy failure, query deployment operations for specific resource error reasons
  • Stack deletion with backoff: Poll deletion status with exponential backoff until confirmed deleted
  • UUID-based test correlation: Generate unique identifiers per test run to isolate assertions
  • Trigger/subscription validation: Verify Event Grid subscriptions and Event Hub bindings are correctly wired after deployment
  • Centralized config: Single .cfg file for all test constants — no hardcoded values in scripts

CI Integration Notes

  • Run on every PR approval (triggered by pull_request_review event with state=approved)
  • Total expected duration: ~20-25 minutes (dominated by flow log generation wait)
  • On failure: preserve Application Insights logs for debugging before teardown
  • Report results to test dashboard with pass/fail and latency metrics
